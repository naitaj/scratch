# PRD: Anti-Gravity Automated Ingestion Pipeline (ConstituencyIQ Phase 1)

## Problem Statement
When executing Phase 1 of a political data intelligence pipeline, the process of manually identifying, downloading, and cataloging election data for 243 separate constituencies is highly inefficient and vulnerable to extensive human error. Government election portals often feature irregular layouts, rely heavily on client-side dynamic content rendering, and display inconsistent naming syntax for constituencies. Attempting to manually gather, rename, and align these records from disconnected sources (ECI reports, Census portals, welfare program statistics, and media platforms) causes a massive operational bottleneck right at the beginning of development, complicating subsequent data cleaning and fuzzy matching steps.

## Goal
The goal is to engineer an automated, robust web scraping and API ingestion pipeline ("Anti-Gravity Data Harvester") that autonomously executes **Phase 1: Raw Data Acquisition** as detailed in `constituencyiq_build_procedure.md`. The pipeline will extract all core datasets for the Bihar 2025 elections, standardizing file taxonomy at the point of download, and store them directly in the target `data/raw/` directories to unblock downstream feature engineering and clustering modules.

## Target User
*   **Primary User**: Shadwal (the data engineer/campaign architect) who requires continuous, uncorrupted, and structured raw data feeds to populate the `constituencyiq` pipeline without manual data handling overhead.

## User Stories
*   As a builder, I want the pipeline to systematically scrape all ECI result files and index cards so that I do not have to perform hundreds of manual file downloads.
*   As a data cleaner, I want files to be saved with a strict, deterministic ID prefix right at download so that I can eliminate time-consuming fuzzy-matching phases on inconsistent constituency names later.
*   As a political analyst, I want candidate asset, liability, and background metadata to be extracted from candidate affidavit documents so I can include critical rival insights within the final intelligence dossier.
*   As an automation operator, I want the pipeline to run autonomously and log failures gracefully so that temporary website crashes do not ruin the entire execution run.

## Functional Requirements

### FR1: ECI Results Data Harvesting Module
*   The system shall parse `results.eci.gov.in` and related statistical portals to isolate the 2025 Bihar Assembly Election results[cite: 1, 2].
*   The system shall prioritize downloading data in structured spreadsheet formats (XLSX) over PDF documents where available.
*   The system shall extract vote distributions (EVM vs. Postal), total polled electors, candidate tallies, margins, and the specific Index Card summaries[cite: 1, 2].

### FR2: Candidate Affidavit Scraper (Burden-Reducer Layer)
*   The system shall target the ECI Candidate Affidavit portal to systematically capture Form 26 documentation for all contesting candidates.
*   The system shall use text extraction logic to parse candidate name, party affiliation, total reported assets, total liabilities, highest education level, and flags for active criminal cases.

### FR3: Primary Census Abstract Ingestion Module
*   The system shall systematically grab district and block-level Census datasets containing demographic indicators (total population, literacy breakdown, rural/urban ratios, SC/ST splits, and occupation mapping)[cite: 1, 2].
*   The system shall download from verified pre-compiled mirrors (such as GitHub or Kaggle data archives) to bypass anti-scraping protections on the primary census server.

### FR4: Welfare Scheme Allocation Tracker
*   The system shall scrape block-level performance tables from `nrega.nic.in` (Bihar State Page), alongside public portals for PMAY and Ujjwala[cite: 2].
*   The system shall collect total active job cards (MGNREGA), home construction sanctions (PMAY), and gas connection volume metrics (Ujjwala)[cite: 1, 2].

### FR5: News & Media Discourse Aggregator
*   The system shall interface with NewsAPI and supplement with targeted parsing of local press feeds (such as the *Hindustan Times* and *The Indian Express*)[cite: 2].
*   The system shall apply a strict publication window filter restricted to the pre-election timeframe of September 1, 2024, to November 30, 2024[cite: 2].
*   The system shall retrieve article title, source publication, publishing date, snippet text, and URL string tagged to specific constituency names[cite: 2].

### FR6: Geographic Coordinates Downloader (Burden-Reducer Layer)
*   The system shall fetch boundary files (GeoJSON/Shapefile formats) representing the spatial polygons of Bihar's 243 assembly constituencies to fulfill downstream map requirements[cite: 1, 2].

### FR7: Standardized Taxonomy Naming Engine
*   The system shall check an internal master ID mapping lookup table during download execution[cite: 2].
*   The system shall force a strict naming format on every file saved to disk using a unique identifier structure, for example: `AC001_valmiki_nagar_eci2025.xlsx`[cite: 2].

## Non-Functional Requirements
*   **NFR1: Anti-Blocking Mechanisms**: The system must incorporate randomized delay pauses (varying dynamically between 2.0 to 5.0 seconds) and cycle HTTP user-agent profiles to circumvent server rate limits.
*   **NFR2: Unattended Execution**: The pipeline must run completely end-to-end without manual intervention, outputting structured trace execution logs directly to an isolated log repository.
*   **NFR3: Structural Validation**: The scraper must perform file integrity verification (e.g., checking for zero-byte payloads or invalid PDF headers) right after download, automatically re-queueing broken items.
*   **NFR4: Directory Integrity**: Downloaded outputs must map explicitly to the predefined system storage tree outline such as `data/raw/eci_2025/` and `data/raw/census/`[cite: 2].

## User Flow
1. The developer populates a master constituency look-up layout spreadsheet with official IDs and structural names.
2. The developer runs the "Anti-Gravity" pipeline via a terminal command or automation trigger.
3. The pipeline orchestrator verifies connectivity, reads the look-up layout, and initializes Modules 1 through 6 sequentially or across controlled concurrent worker threads.
4. The harvesting engines query the targeted portals, navigate client-side navigation structures, extract datasets, assign the standardized prefix titles, and dump the files into designated `data/raw/` directories[cite: 2].
5. On pipeline completion, the orchestrator outputs a data acquisition completion status log file.

## Design Requirements
*   **DR1: Standardized Storage Mapping**: Data storage layout must directly mirror Phase 0 instructions specified within `constituencyiq_build_procedure.md`[cite: 2]:
data/raw/
  ├── eci_2025/
  ├── eci_2020/
  ├── candidate_affidavits/
  ├── census/
  ├── schemes/
  ├── news/
  └── spatial/
```
DR2: Log Visibility: Terminal telemetry must clearly distinguish processing alerts with standardized coloring patterns: [SUCCESS] in green, [WARN] in yellow for minor data gaps, and [ERROR] in red for system connection timeouts.

DR3: Structural Formats: Tabular statistics parsed directly from web layers must be exported cleanly into standardized JSON Lines (.jsonl) or plain CSV formats to ensure simple parsing.

Technical Constraints
TC1: Client-Side Rendering (DOM Domination): Basic static HTTP scraping via libraries like BeautifulSoup will fail on dynamic government portals[cite: 2]. The pipeline must execute via programmatic automation tools like Playwright or Selenium running headless to interpret JavaScript structures.

TC2: Rate and Payload Restrictions: News harvesting modules must track API query consumption carefully to stay within standard free-tier service thresholds without triggering connection restrictions[cite: 2].

TC3: Administrative Boundary Mapping: Census variables are anchored to old 2011 administrative configurations, requiring a cross-referenced matching matrix to align smoothly with 2025 assembly bounds[cite: 1, 2].

Edge Cases
EC1: Mid-Scrape Portal Failures: If a target site drops connection during scraping, the pipeline must log the error, cache existing elements, and allow execution to resume from the last successful constituency ID checkpoint without pulling redundant duplicates.

EC2: Unreadable PDF Documentation: If a candidate affidavit file contains scanned imagery unreadable by typical parsers, the system must save the physical document to disk, flag the exception within execution files, and skip text generation for that record instead of halting the engine.

EC3: Multi-Word Naming Variability: Discrepancies like "Valmiki Nagar" vs. "Valmikinagar" must be tokenized, converted to lowercase, stripped of white spaces, and resolved back to the primary key name (valmiki_nagar) dynamically[cite: 2].

Acceptance Criteria
AC1: The automated script runs unattended and fetches electoral history, demographic profiles, welfare indicators, and news details for all 243 assembly constituencies in Bihar[cite: 1, 2].

AC2: 100% of the raw downloads match the unified identifier taxonomy matrix, dropping manual renaming workflows during subsequent joining phases[cite: 2].

AC3: Automatic background validations successfully capture and re-download broken, incomplete, or empty files before ending the execution flow.

AC4: The harvesting engine builds an accurate, preliminary draft structure for data_dictionary.md, tracing file system paths and total record metrics to meet Phase 1 closure criteria[cite: 2].

Out of Scope
OS1: Frontend Interface Execution: Developing the visual React dashboard, map components, or analytics layouts[cite: 1, 2].

OS2: Advanced Feature Processing: Running text embeddings (all-MiniLM-L6-v2), execution of BERTopic algorithms, or computing K-Means clustering profiles which are mapped to later phases[cite: 2].

OS3: Historical Election Context Expansion: Extracting historical datasets from election rounds taking place before the 2020 base timeline[cite: 1, 2].
